# 下一次纯净重装操作清单

版本：3.0.0-rc2 草稿  
适用：DMIT、Debian 12/13、`senyz.top`、VMess + WebSocket + TLS、Cloudflare 官方 WARP、macOS/iPhone Shadowrocket

## 先看这一段

正常重装只走一条主线：

1. 自己生成 SSH 密钥，只把公钥交给 DMIT。
2. 把 `senyz.top` 的 DNS-only A 记录改为新服务器 IPv4，不设置 AAAA。
3. 在 DMIT 重装 Debian 13，并选择自己的公钥。
4. SSH 登录后粘贴一次“固定版本安装命令”。
5. 按提示确认第二个 SSH 窗口和 Serial Console；若 WARP 注册临时受限，以后只需运行同一条继续命令。
6. 脚本最后直接显示 Shadowrocket 导入信息和二维码。

脚本内部仍分“基础代理”和“WARP”两阶段，是为了在 WARP 失败时保住 SSH 和明确停止代理直连；你不需要分别运行或理解这些子脚本。

## 一、重装前只做一次：准备自己的 SSH 密钥

### 当前使用 Windows 时

在 PowerShell 中执行：

```powershell
New-Item -ItemType Directory -Force "$HOME\.ssh" | Out-Null
ssh-keygen -t ed25519 -a 100 -f "$HOME\.ssh\dmit_senyz_ed25519" -C "dmit-senyz"
icacls "$HOME\.ssh\dmit_senyz_ed25519" /inheritance:r
icacls "$HOME\.ssh\dmit_senyz_ed25519" /grant:r "$($env:USERNAME):(R)"
Get-Content "$HOME\.ssh\dmit_senyz_ed25519.pub"
```

`ssh-keygen` 会要求输入两次密钥口令。请设置一个能记住的强口令。最后一条命令显示的是公钥，可以粘贴到 DMIT；不要上传或发送没有 `.pub` 后缀的私钥。

原来的 `id_rsa.pem` 暂时保留，等新服务器、Shadowrocket 和控制台全部验证正常后再决定是否归档。

### 以后使用 macOS 时

在“终端”中执行：

```bash
mkdir -p ~/.ssh
chmod 700 ~/.ssh
ssh-keygen -t ed25519 -a 100 -f ~/.ssh/dmit_senyz_ed25519 -C dmit-senyz
chmod 600 ~/.ssh/dmit_senyz_ed25519
cat ~/.ssh/dmit_senyz_ed25519.pub
```

把私钥做一份加密备份。DMIT 和 GitHub 中只能出现公钥，不能出现私钥。

## 二、DMIT 与 DNS

### 1. 在 DMIT 保存公钥

在 DMIT 控制面板的 SSH Keys 页面新建密钥，名称可写 `senyz-mac-ed25519` 或 `senyz-windows-ed25519`，内容粘贴上一节显示的整行 `.pub` 公钥。

这不是让 DMIT 替你生成私钥。私钥始终只保存在自己的电脑和加密备份中。

### 2. 重装前记录新 IPv4

如果重装后服务器 IPv4 不变，继续使用原记录；如果更换了服务器或 IPv4，先记录新地址。

### 3. 修改 Cloudflare DNS

`senyz.top` 使用下面的设置：

| 项目 | 值 |
|---|---|
| 类型 | A |
| 名称 | `@` |
| 内容 | 新服务器 IPv4 |
| 代理状态 | DNS only（灰色云朵） |
| TTL | 300 或 Auto |
| AAAA | 不创建；已有则删除 |

不需要 Cloudflare Global API Key、API Token 或 DNS API。`senyz.art` 暂时保留，不参与这次迁移，避免同时改变服务器、域名和协议。

Windows 可用下面两条只读命令确认：

```powershell
Resolve-DnsName senyz.top -Type A
Resolve-DnsName senyz.top -Type AAAA
```

第一条应显示新 IPv4；第二条应没有 AAAA 结果。DNS 尚未更新时不要运行服务器安装脚本。

### 4. 在 DMIT 重装

选择 Debian 13，并在重装页面选择刚刚上传的自己的 SSH 公钥。不要选择“由 DMIT 新生成并下载私钥”的流程。

等待重装完成后，打开 DMIT Serial Console 并按一次 Enter。此时只需确认能看到类似 `login:` 的登录提示，**不需要登录，也不要为了这一步提前设置 root 密码**。引导脚本完成基础阶段后会让你设置一个只供控制台救援使用的密码，再让你实际登录验证。

Serial Console 用于在 SSH 或网络故障时直接进入 Linux 文本控制台。VNC Console 显示虚拟显示器和启动画面，主要用于系统启动、网络完全失效或 Serial Console 无法交互时的救援；日常维护优先用 SSH，救援优先用 Serial Console。

## 三、第一次 SSH 登录

同一个 IPv4 重装后，旧主机指纹会变化。只删除这一台服务器的旧记录：

Windows：

```powershell
ssh-keygen -R <服务器IPv4>
ssh -i "$HOME\.ssh\dmit_senyz_ed25519" root@<服务器IPv4>
```

macOS：

```bash
ssh-keygen -R <服务器IPv4>
ssh -i ~/.ssh/dmit_senyz_ed25519 root@<服务器IPv4>
```

不要清空整个 `known_hosts`。有条件时，可先在 DMIT Serial Console 中执行下面的命令，再与 SSH 首次连接显示的 ED25519 指纹比较：

```bash
ssh-keygen -lf /etc/ssh/ssh_host_ed25519_key.pub
```

## 四、粘贴一次安装命令

下面使用已经回读核对过的不可变提交和 SHA256；即使草稿分支以后继续修改，这个入口下载的仍是本次审核版本：

```bash
SETUP_COMMIT="fc2de6c159b265721d76555ddbc6f7d6e908309d"
SETUP_SHA256="92985dec5d2692c0aa7934d9b6cc5da8848ba5a4d74c311616d70bdbfc833ad2"
RAW="https://raw.githubusercontent.com/syubroken/server_proxy_setup"
curl --proto '=https' --tlsv1.2 -fsSLo setup_script.sh \
  "${RAW}/${SETUP_COMMIT}/setup_script.sh"
echo "${SETUP_SHA256}  setup_script.sh" | sha256sum -c -
chmod 700 setup_script.sh
./setup_script.sh
```

依次按屏幕提示：

1. 域名直接按 Enter，使用默认的 `senyz.top`。
2. 输入接收证书通知的邮箱；它不会写入 GitHub。
3. 基础部署确认时输入大写 `YES`。
4. 基础阶段完成后，保留当前窗口并打开第二个 SSH 窗口，确认自己的密钥可以登录。
5. 回到第一个窗口，脚本要求设置控制台 root 密码时输入两次新密码。这个密码只供 Serial/VNC 本地救援，SSH 密码登录仍然关闭。
6. 这时才在 DMIT Serial Console 用 `root` 和刚设置的密码登录。
7. 两处都成功后，回到第一个 SSH 窗口输入大写 `READY`。

脚本随后会自动安装 Cloudflare 官方 WARP、使用 MASQUE、验证 IPv4/IPv6 均为 WARP 出站、启动故障保护并执行最终只读验收。WARP 注册只尝试一次，不会遇到限流后每几秒循环请求。成功后直接显示客户端参数和二维码。

## 五、导入 Shadowrocket

客户端链接属于凭据，不要放进 GitHub、普通笔记、聊天或公开截图。

### macOS

1. 复制脚本最后显示的整行 `vmess://...` 链接。
2. 在 Shadowrocket 中选择从剪贴板导入。
3. 选择新节点，将全局路由设为“代理”，再开启连接。
4. 先访问普通网站确认网络，再登录重要账号。

### iPhone

1. 用 Shadowrocket 扫描脚本最后显示的二维码，或从剪贴板导入链接。
2. 选择新节点，将全局路由设为“代理”。
3. 允许 iOS 添加 VPN 配置并开启连接。

以后需要再次显示参数时，只执行这一条容易记的命令：

```bash
senyz-show-client
```

再次显示二维码：

```bash
senyz-show-client qr
```

不需要手动寻找 `/root` 文件，也不需要处理“权限 600”；脚本已经在后台完成保护。

## 六、完成后怎样判断成功

安装器已经自动验收。以后每月或感觉异常时，只执行：

```bash
senyz-verify-rebuild --require-warp
```

最后显示 `Result: PASS` 即可。它不会打印 UUID、WebSocket 路径、客户端链接或私钥。

证书由 Certbot Webroot 和 systemd timer 自动续期，不需要按到期日手工重装。只有验收显示证书、Certbot timer 或 Nginx 为 `FAIL` 时，再保存输出并排查。

## 七、失败时不要猜

### 在输入 READY 以前失败

WARP 尚未改变路由。保留 SSH 窗口，把 `/root/senyz-base-rebuild.log` 和下面命令的输出用于排查：

```bash
senyz-verify-rebuild
```

日志和验收不会主动打印客户端链接，但发送前仍应检查是否包含邮箱或其他不想公开的信息。

### 输入 READY 以后，出现 `HTTP 429` 或 `Too Many Requests`

这是 Cloudflare 的 WARP 消费者注册接口临时限流，不是 DNS 设置错误，也不是因为没有填写 Cloudflare Global API Key。新版只请求一次并暂停，不会开始切换 WARP 路由，V2Ray 保持运行。

此时不要重装 Debian，也不要连续重试。等待一段时间后只运行：

```bash
/root/senyz-finish-rebuild
```

Cloudflare 没有为这种情况提供可靠的固定等待时长；如果稍后仍然显示 429，就隔天再运行同一条命令。引导器会自动识别已经完成的基础阶段和控制台检查，不会重新生成节点、再次询问邮箱或重新设置控制台密码。

在最终显示完整验收 `Result: PASS` 前，基础代理尚未获得 WARP 出口，不要用它登录要求保持 WARP 出口的重要账号。

### WARP 已开始切换路由后失败

脚本会保持 V2Ray 停止，避免客户端悄悄改用 DMIT 原始出口。第二个 SSH 和 DMIT Console 应仍可使用。先运行同一个继续命令：

```bash
/root/senyz-finish-rebuild
```

如果再次失败，再保存下面两条输出用于排查：

```bash
/root/senyz-warp.sh status
/root/senyz-warp.sh logs
```

不要为了“先能用”而关闭 fail-closed。可以把这两段无凭据输出交给 Codex 分析；确需修复时再执行脚本建议的操作。只有继续入口和排查都无法恢复时，才考虑重装并使用保留的旧流程。

## 八、本次实测发现了什么

1. `3.0.0-rc1` 用 `getent ahostsv6` 检查 AAAA。Debian 把服务器 IPv4 表示为 `::ffff:IPv4` 时，脚本误以为域名存在真实 AAAA 并停止。`rc2` 改为用 `dig` 查询 DNS 中真正发布的 AAAA；Cloudflare 只有 A 记录时不会再误报。
2. 旧 `git.io/warp.sh` 在 `wgcf-account.toml` 没生成时，每 5 秒重新运行一次注册，没有针对 429 的退避和次数上限。它依赖的 `wgcf` 注册失败后会留下已安装的软件包，但没有清楚的续跑状态。
3. 原始旧脚本读取并导出了 Cloudflare API Key，但证书实际采用 standalone 验证；后面的 `wgcf register` 是另一套 Cloudflare 消费者注册接口。因此留空 Global API Key 与这次 429 没有因果关系。
4. 官方 `warp-cli registration new` 也要创建消费者 WARP 注册，所以不能保证永远不出现 429。`rc2` 解决的是安全暂停、停止高频重试和一条命令续跑，不是假装能够绕过 Cloudflare 的限流。

## 九、长期维护和边界

- Debian 安全更新会自动安装，但不会自动重启。验收或系统提示需要重启时，先确认 Serial Console 和第二个 SSH 窗口可用。
- WARP 只改善服务器出站路径和地区呈现，不能保证任何 AI 服务账号不受限制，也不能改变 DMIT 入站 IPv4被干扰的可能性。
- 使用重要账号时尽量保持同一服务器、同一 WARP 出站和同一客户端节点，不在会话中频繁切换线路。
- 一次只改变一个大变量。本次继续用 `senyz.top`、VMess + WebSocket + TLS；`senyz.art` 和其他协议留作以后独立评估。
- 不再执行 `bash <(curl -fsSL git.io/warp.sh)`。旧脚本只作为明确的灾难恢复备份，不是新流程的一部分。
- 仓库中的 `legacy/` 在这次整理中保持原内容和原执行行为不变。
